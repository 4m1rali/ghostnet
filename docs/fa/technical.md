## DPI چیست؟

بازرسی عمیق بسته (Deep Packet Inspection یا DPI) یک فناوری فیلترینگ شبکه است که توسط اپراتورهای اینترنت، دولت‌ها و فایروال‌های سازمانی استفاده می‌شود. این فناوری نه‌تنها آدرس مبدا و مقصد ترافیک، بلکه محتوای واقعی بسته‌های شبکه را نیز بررسی می‌کند.

### چگونه DPI در لایه TLS کار می‌کند

وقتی به یک وب‌سایت از طریق HTTPS (پورت ۴۴۳) متصل می‌شوید، مرورگر شما یک بسته TLS ClientHello ارسال می‌کند. این بسته حاوی فیلدی به نام **Server Name Indication (SNI)** است — نام هاست وب‌سایتی که به آن متصل می‌شوید. SNI قبل از برقراری تونل رمزنگاری‌شده، به صورت متن ساده ارسال می‌شود.

سیستم‌های DPI این بسته را رهگیری کرده و فیلد SNI را می‌خوانند. اگر نام هاست در لیست سیاه باشد، اتصال قطع می‌شود. اگر در لیست سفید (دامنه‌های مجاز) باشد، اتصال برقرار می‌شود.

```
مرورگر → [TLS ClientHello: SNI="سایت-مسدود.com"] → DPI → مسدود ✗
مرورگر → [TLS ClientHello: SNI="hcaptcha.com"]    → DPI → مجاز ✓
```

### چرا SNI قابل مشاهده است

TLS داده‌های برنامه را رمزنگاری می‌کند، اما مذاکره handshake قبل از برقراری رمزنگاری انجام می‌شود. پسوند SNI (RFC 6066) برای این طراحی شده که یک سرور بتواند چندین گواهی TLS داشته باشد — سرور باید بداند کدام گواهی را ارائه دهد قبل از اینکه کانال رمزنگاری‌شده وجود داشته باشد.

## چگونه GhostNet از DPI عبور می‌کند

GhostNet از یک تفاوت اساسی بین نحوه پردازش بسته‌ها توسط سیستم‌های DPI و سرورهای TCP بهره می‌برد.

**سیستم‌های DPI بدون حالت (stateless) هستند** — بسته‌ها را بدون نگهداری وضعیت کامل TCP بررسی می‌کنند. آن‌ها SNI را از اولین بسته داده‌دار می‌خوانند و تصمیم می‌گیرند.

**سرورهای TCP با حالت (stateful) هستند** — یک پنجره دریافت نگهداری می‌کنند و بسته‌هایی که خارج از آن هستند را دور می‌اندازند.

GhostNet یک TLS ClientHello جعلی با SNI مجاز را *قبل* از ClientHello واقعی تزریق می‌کند، با استفاده از یک شماره توالی که بسته را خارج از پنجره دریافت سرور قرار می‌دهد. سیستم DPI SNI مجاز را می‌بیند و اتصال را مجاز می‌داند. سرور بسته جعلی را بی‌صدا دور می‌اندازد. ClientHello واقعی به دنبال آن می‌آید و اتصال به طور عادی برقرار می‌شود.

### جریان کامل بسته‌ها

```
کلاینت       GhostNet              DPI              سرور هدف
  │              │                   │                    │
  │── اتصال ────►│                   │                    │
  │              │── SYN ───────────►│───────────────────►│
  │              │◄─ SYN-ACK ────────│◄───────────────────│
  │              │── ACK ───────────►│───────────────────►│
  │              │                   │                    │
  │              │   [GhostNet بسته جعلی تزریق می‌کند]    │
  │              │── PSH+ACK ────────►│                   │
  │              │   seq = ISN+1-N    │                   │
  │              │   SNI = "hcaptcha.com" (مجاز)          │
  │              │                   │                    │
  │              │   DPI SNI را می‌خواند:                  │
  │              │   "hcaptcha.com"  │                    │
  │              │   → مجاز ✓        │                    │
  │              │                   │  سرور دور می‌اندازد│
  │              │                   │  (خارج از پنجره)   │
  │              │── ClientHello واقعی ─────────────────►│
  │              │◄──────────────────────── ServerHello ──│
  │◄═════════════ رله دوطرفه ════════════════════════════►│
```

## استراتژی‌های دور زدن

### ۱. wrong_seq (اصلی)

تکنیک اصلی توضیح داده شده در بالا. سازگارترین روش — در برابر اکثر استقرارهای DPI کار می‌کند.

**محاسبه شماره توالی:**
```
fake_seq = (ISN + 1 - len(payload)) & 0xFFFFFFFF
```

### ۲. fragment (تکه‌تکه کردن)

ClientHello جعلی را در چندین بخش TCP تقسیم می‌کند. نقطه تقسیم بایت ۱ است — فقط بایت نوع رکورد TLS (`0x16`).

**چرا کار می‌کند:** بسیاری از سیستم‌های DPI فقط اولین بخش TCP را بررسی می‌کنند. یک بایت (`0x16`) یک رکورد TLS کامل نیست — DPI نمی‌تواند SNI را از آن تجزیه کند.

### ۳. desync — checksum بد

یک بسته با checksum TCP عمداً خراب ارسال می‌کند.

**چرا کار می‌کند:** برخی سیستم‌های DPI بدون حالت checksum‌های TCP را اعتبارسنجی نمی‌کنند. سرور هدف همیشه checksum‌ها را اعتبارسنجی می‌کند و بسته را بی‌صدا دور می‌اندازد.

### ۴. desync — TTL محدود

ClientHello جعلی را با `TTL = 1` ارسال می‌کند.

**چرا کار می‌کند:** بسته توسط اولین روتر دور انداخته می‌شود. سیستم‌های DPI معمولاً نزدیک‌تر به کلاینت هستند، بنابراین بسته را قبل از انقضا می‌بینند.

### ۵. adaptive (توصیه شده)

استراتژی‌ها را به ترتیب امتحان می‌کند و برنده را برای هر `(dstIP, dstPort)` کش می‌کند:

```
wrong_seq → fragment → desync_bad_cs → desync_ttl
```

## اثر انگشت TLS

GhostNet رکوردهای TLS ClientHello کامل و معتبر می‌سازد که دقیقاً با اثر انگشت مرورگرهای واقعی مطابقت دارند.

| پروفایل | مبتنی بر | GREASE | compress_cert |
|---|---|---|---|
| `chrome` | Chrome 124 | ✓ | ✓ |
| `firefox` | Firefox 125 | ✗ | ✗ |
| `safari` | Safari 17 | ✗ | ✗ |
| `edge` | Edge 124 | ✓ | ✓ |
| `chrome120` | Chrome 120 | ✓ | ✗ |
| `random` | هر اتصال متفاوت | — | — |

## مدل زمان‌بندی

تأخیر تزریق از یک مدل احتمالاتی دو مؤلفه‌ای استفاده می‌کند:

**مؤلفه گاوسی (۹۲٪ اتصالات):**
```
تأخیر = base_ms + N(0, σ)    که σ = base_ms × 0.30
```

**دنباله Weibull (۸٪ اتصالات):**
```
تأخیر = λ × (-ln(1-U))^(1/k) + 5ms
که λ=8, k=1.5, U ~ Uniform(0,1)
```

این مدل رفتار واقعی مرورگر را شبیه‌سازی می‌کند و تشخیص مبتنی بر زمان‌بندی را دشوار می‌کند.

## تزریق سوکت خام

GhostNet بسته‌های کامل IPv4 + TCP را به صورت دستی می‌سازد و از طریق `AF_INET SOCK_RAW` با `IP_HDRINCL` تزریق می‌کند. این نیاز به `CAP_NET_RAW` در لینوکس دارد.

### جعل TTL

وقتی `ttl_spoof` فعال است:
```
base = انتخاب تصادفی از {64, 128}
hops = عدد صحیح تصادفی در [1, 8]
TTL  = base - hops
```

این باعث می‌شود بسته تزریق‌شده به نظر برسد از یک هاست واقعی در فاصله شبکه‌ای می‌آید.
