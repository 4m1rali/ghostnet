# Техническое руководство GhostNet

## Что такое глубокая инспекция пакетов (DPI)?

Глубокая инспекция пакетов (Deep Packet Inspection, DPI) — технология сетевой фильтрации, используемая интернет-провайдерами, правительствами и корпоративными межсетевыми экранами. Она анализирует не только адреса источника и назначения трафика, но и фактическое содержимое сетевых пакетов.

### Как DPI работает на уровне TLS

При подключении к веб-сайту через HTTPS (порт 443) браузер отправляет пакет TLS ClientHello. Этот пакет содержит поле **Server Name Indication (SNI)** — имя хоста сайта, к которому вы подключаетесь. SNI передаётся в открытом виде до установления зашифрованного туннеля.

Системы DPI перехватывают этот пакет и читают поле SNI. Если имя хоста находится в чёрном списке — соединение блокируется. Если в белом списке — разрешается.

```
Браузер → [TLS ClientHello: SNI="заблокированный-сайт.com"] → DPI → БЛОК ✗
Браузер → [TLS ClientHello: SNI="hcaptcha.com"]              → DPI → РАЗРЕШЕНО ✓
```

### Почему SNI виден

TLS шифрует данные приложения, но согласование handshake происходит до установления шифрования. Расширение SNI (RFC 6066) разработано для того, чтобы один сервер мог хранить несколько TLS-сертификатов — сервер должен знать, какой сертификат предоставить, до создания зашифрованного канала.

## Как GhostNet обходит DPI

GhostNet использует фундаментальное различие между тем, как системы DPI и TCP-серверы обрабатывают пакеты.

**Системы DPI не имеют состояния (stateless)** — они проверяют пакеты по мере поступления без поддержания полного состояния TCP. Они читают SNI из первого пакета с данными и принимают решение.

**TCP-серверы имеют состояние (stateful)** — они поддерживают окно приёма и отбрасывают пакеты, выходящие за его пределы.

GhostNet внедряет поддельный TLS ClientHello с разрешённым SNI *перед* настоящим, используя порядковый номер, который помещает пакет за пределы окна приёма сервера. Система DPI видит разрешённый SNI и пропускает соединение. Сервер молча отбрасывает поддельный пакет. Настоящий ClientHello следует за ним, и соединение устанавливается нормально.

### Полный поток пакетов

```
КЛИЕНТ       GhostNet              DPI              ЦЕЛЕВОЙ СЕРВЕР
  │              │                   │                    │
  │── connect ──►│                   │                    │
  │              │── SYN ───────────►│───────────────────►│
  │              │◄─ SYN-ACK ────────│◄───────────────────│
  │              │── ACK ───────────►│───────────────────►│
  │              │                   │                    │
  │              │   [GhostNet внедряет поддельный пакет] │
  │              │── PSH+ACK ────────►│                   │
  │              │   seq = ISN+1-N    │                   │
  │              │   SNI = "hcaptcha.com" (разрешено)     │
  │              │                   │                    │
  │              │   DPI читает SNI: │                    │
  │              │   "hcaptcha.com"  │                    │
  │              │   → РАЗРЕШЕНО ✓   │                    │
  │              │                   │  сервер отбрасывает│
  │              │                   │  (вне окна)        │
  │              │── настоящий ClientHello ──────────────►│
  │              │◄──────────────────────── ServerHello ──│
  │◄═════════════ двунаправленный ретранслятор ══════════►│
```

## Стратегии обхода

### 1. wrong_seq (основная)

Основная техника, описанная выше. Наиболее совместимая — работает против большинства развёртываний DPI.

**Вычисление порядкового номера:**
```
fake_seq = (ISN + 1 - len(payload)) & 0xFFFFFFFF
```

### 2. fragment (фрагментация)

Разбивает поддельный ClientHello на несколько TCP-сегментов. Точка разбиения — байт 1 (только байт типа TLS-записи `0x16`).

**Почему работает:** Многие системы DPI проверяют только первый TCP-сегмент. Один байт (`0x16`) — не полная TLS-запись, DPI не может извлечь из него SNI.

### 3. desync — плохая контрольная сумма

Отправляет пакет с намеренно испорченной контрольной суммой TCP.

**Почему работает:** Некоторые stateless DPI-системы не проверяют контрольные суммы TCP. Целевой сервер всегда проверяет контрольные суммы и молча отбрасывает пакет.

### 4. desync — ограниченный TTL

Отправляет поддельный ClientHello с `TTL = 1`.

**Почему работает:** Пакет отбрасывается первым маршрутизатором. Системы DPI обычно расположены ближе к клиенту, чем первый маршрутизатор, поэтому они видят пакет до его истечения.

### 5. adaptive (рекомендуется)

Пробует стратегии по порядку и кэширует победителя для каждой пары `(dstIP, dstPort)`:

```
wrong_seq → fragment → desync_bad_cs → desync_ttl
```

## Отпечаток TLS

GhostNet строит полные, валидные TLS 1.3 ClientHello-записи с нуля, точно соответствующие отпечаткам реальных браузеров.

| Профиль | Основан на | GREASE | compress_cert |
|---|---|---|---|
| `chrome` | Chrome 124 | ✓ | ✓ |
| `firefox` | Firefox 125 | ✗ | ✗ |
| `safari` | Safari 17 | ✗ | ✗ |
| `edge` | Edge 124 | ✓ | ✓ |
| `chrome120` | Chrome 120 | ✓ | ✗ |
| `random` | Меняется каждое соединение | — | — |

## Модель задержки

Задержка внедрения использует двухкомпонентную вероятностную модель:

**Гауссова компонента (92% соединений):**
```
задержка = base_ms + N(0, σ)    где σ = base_ms × 0.30
```

**Хвост Вейбулла (8% соединений):**
```
задержка = λ × (-ln(1-U))^(1/k) + 5ms
где λ=8, k=1.5, U ~ Uniform(0,1)
```

Эта модель имитирует реальное поведение браузера и затрудняет обнаружение по временны́м паттернам.

## Внедрение через raw-сокет

GhostNet вручную конструирует полные пакеты IPv4 + TCP и внедряет их через `AF_INET SOCK_RAW` с `IP_HDRINCL`. Требует `CAP_NET_RAW` в Linux.

### Подмена TTL

При включённом `ttl_spoof`:
```
base = случайный выбор из {64, 128}
hops = случайное целое в [1, 8]
TTL  = base - hops
```

Это делает внедрённый пакет похожим на пакет от реального хоста на сетевом расстоянии.
